Lightbox Approach Options

Three ways to implement the photo popup — pick the one that feels right

A

Pure CSS + Vanilla JS (inline)

Self-contained within PhotoGallery.astro. No dependencies. A dark overlay fills the viewport; the image is centered with prev/next arrow buttons on the sides and an ✕ close button in the top-right corner. Keyboard navigation (←/→/Esc) included. Caption shown below image.

Pros

  • Zero new dependencies
  • Fastest to ship
  • Fits the existing Astro + Tailwind stack perfectly

Cons

  • More code inside the component
B

Separate LightboxModal component

Extract the overlay into its own LightboxModal.astro component, keeping PhotoGallery.astro lean. The gallery renders thumbnails; the modal handles display, navigation, and accessibility. Shared via a tiny global JS event bus (openLightbox(photos, index)).

Pros

  • Clean separation of concerns
  • Modal reusable across other galleries

Cons

  • Slightly more files
C

Third-party lightbox library (e.g. GLightbox)

Drop in a polished library like GLightbox or PhotoSwipe via npm. Handles animations, touch swipe, zoom, and accessibility out of the box. Wire it up with a data-gallery attribute on the existing anchors.

Pros

  • Feature-rich (swipe, zoom, animations)
  • Battle-tested accessibility

Cons

  • Adds a JS dependency
  • External library styling may clash with brand

💡 Recommendation: B — separate component keeps things clean and the modal stays reusable if you add photo galleries elsewhere on the site.
